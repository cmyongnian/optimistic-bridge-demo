# Optimistic Bridge Demo

本项目是一个教学版乐观桥复现实验，用于展示区块链跨链桥中的信任最小化机制。

项目并不实现真实资产跨链，也不部署到真实区块链网络，而是在 Hardhat 本地环境中使用两个智能合约模拟源链和目标链之间的跨链消息传递过程。

## 1. 项目目标

本项目的目标是复现乐观桥中的核心流程：

1. 用户在源链发起跨链消息；
2. 源链合约生成消息哈希；
3. 中继者将消息提交到目标链；
4. 目标链先将消息标记为待确认状态；
5. 系统进入挑战期；
6. 观察者可以在挑战期内挑战错误消息；
7. 如果无人挑战，消息在挑战期结束后被最终确认。

该实验用于说明乐观桥的核心思想：

> 目标链不立即完全相信中继者提交的消息，而是通过挑战窗口和观察者机制降低对第三方的信任依赖。

## 4. 合约说明

### SourceBridge.sol

`SourceBridge` 用于模拟源链桥合约。

它的主要作用是：

- 接收用户发起的跨链消息；
- 为每条消息生成 nonce；
- 使用 `keccak256` 计算消息哈希；
- 记录消息哈希；
- 通过事件通知外部中继者。

核心函数：

```solidity
sendMessage(address to, uint256 amount, string calldata data)
```

### TargetBridge.sol

`TargetBridge` 用于模拟目标链桥合约。

它的主要作用是：

- 接收中继者提交的跨链消息；
- 将消息标记为 Pending；
- 设置挑战期；
- 允许观察者在挑战期内挑战错误消息；
- 在挑战期结束后允许消息最终确认。

核心函数：

```solidity
submitMessage(...)
challengeMessage(...)
finalizeMessage(...)
```

## 5. 消息状态

目标链中的消息包含四种状态：

```text
0 = None
1 = Pending
2 = Challenged
3 = Finalized
```

含义如下：

| 状态       | 含义                             |
| ---------- | -------------------------------- |
| None       | 消息不存在                       |
| Pending    | 消息已提交，正在挑战期内等待确认 |
| Challenged | 消息被观察者挑战成功             |
| Finalized  | 挑战期结束后，消息被最终确认     |

## 6. 实验设计

本项目共设计 6 个测试用例。

### 实验一：正常消息在挑战期结束后可以被最终确认

中继者提交的消息与源链消息一致。挑战期结束后，由于无人挑战，消息从 `Pending` 变为 `Finalized`。

该实验验证了乐观桥的正常运行流程。

### 实验二：恶意中继者提交错误消息，观察者可以发起挑战

源链真实消息中的金额为 `100`，但恶意中继者在目标链提交的金额为 `999`。观察者读取源链真实消息哈希后，发现目标链提交内容不一致，于是发起挑战。

挑战成功后，消息状态变为 `Challenged`。

该实验验证了挑战机制可以阻止错误消息。

### 实验三：如果观察者缺席，错误消息也可能被最终确认

恶意中继者提交错误消息，但没有观察者发起挑战。挑战期结束后，错误消息仍然被最终确认。

该实验说明乐观桥并不是无条件安全的，其安全性依赖至少一个诚实观察者在挑战期内持续监控系统。

### 边界测试一：挑战期未结束时，消息不能被最终确认

消息刚提交后，系统仍处于挑战期。如果此时直接调用 `finalizeMessage`，交易会失败。

该测试说明目标链不能立即确认消息，必须等待挑战窗口结束。

### 边界测试二：挑战期结束后，观察者不能再发起挑战

如果观察者在挑战期结束后才尝试挑战，交易会失败。

该测试说明挑战机制具有时间窗口限制。

### 边界测试三：被挑战成功的消息不能再被最终确认

当错误消息已经被观察者挑战成功后，即使挑战期结束，也不能再被最终确认。

该测试说明挑战机制能够阻止恶意消息最终生效。

## 8. 信任最小化分析

传统跨链桥往往依赖中继者、多签委员会或验证者集合来确认跨链消息。如果这些第三方作恶，目标链可能接受错误消息。

本项目复现的乐观桥不要求目标链立即完全相信中继者，而是引入挑战窗口和观察者机制。中继者提交消息后，消息先进入 `Pending` 状态。在挑战期内，观察者可以比较源链真实消息哈希和目标链提交的消息哈希。如果发现不一致，观察者可以发起挑战，使错误消息进入 `Challenged` 状态。

因此，乐观桥的安全性不再依赖单个中继者的诚实，而是转化为以下假设：

> 只要至少存在一个诚实观察者，并且该观察者能在挑战期内发现并挑战错误消息，系统就能阻止恶意消息最终确认。

这体现了信任最小化的思想：系统并不是完全消除信任，而是减少对单一第三方的信任，并通过公开验证、挑战机制和时间窗口提高安全性。

## 9. 局限性

本项目是教学版复现，仍然存在以下局限：

1. 没有实现真实双链通信；
2. 没有实现资产锁定和释放；
3. 没有实现中继者押金和惩罚机制；
4. 没有实现观察者奖励机制；
5. 没有连接真实区块链网络；
6. 没有实现轻客户端验证或零知识证明验证。

因此，本项目重点不是构建可上线的跨链桥，而是复现乐观桥中最核心的信任最小化机制。

## 10. 总结

本实验通过 `SourceBridge` 和 `TargetBridge` 两个合约模拟了乐观桥的基本流程。实验结果表明，挑战窗口和观察者机制能够在一定程度上减少对中继者的信任依赖。

同时，实验三也说明，如果观察者缺席，错误消息仍可能被最终确认。因此，乐观桥的核心优势是降低信任假设，而不是完全消除信任。
